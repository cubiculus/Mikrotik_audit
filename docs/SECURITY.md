# Политика безопасности

## 🔒 Сообщить об уязвимости

Если вы обнаружили уязвимость безопасности, пожалуйста, сообщите нам ответственно.

### ⚠️ Важно

- **НЕ создавайте публичный Issue** для уязвимостей безопасности
- Отправляйте информацию на почту или через private disclosure

## Как сообщить

### Вариант 1: GitHub Security Advisory (рекомендуется)

1. Перейдите на вкладку **Security** репозитория
2. Нажмите **Report a vulnerability**
3. Заполните форму

### Вариант 2: Email

Отправьте описание уязвимости с пометкой "[SECURITY]" в теме.

## Что включать в отчёт

- Тип уязвимости
- Затронутые версии
- Шаги для воспроизведения
- Влияние уязвимости
- Возможное решение (если есть)

## Процесс ответа

1. **Подтверждение** — в течение 48 часов
2. **Оценка** — в течение 5 дней
3. **Исправление** — в зависимости от критичности
4. **Публикация** — после выпуска патча

## Критичность и сроки ответа

| Уровень | Описание | Срок исправления |
|---------|----------|------------------|
| 🔴 Critical | Удалённое выполнение кода, утечка паролей | 24-48 часов |
| 🟠 High | Несанкционированный доступ | 3-5 дней |
| 🟡 Medium | Частичная утечка информации | 1-2 недели |
| 🟢 Low | Минимальное влияние | 2-4 недели |

## Поддерживаемые версии

| Версия | Поддержка |
|--------|-----------|
| 2.x    | ✅ Да     |
| 1.x    | ❌ Нет    |

## Безопасное использование

### Рекомендации

1. **Храните пароли в .env**
   ```bash
   # Правильно
   cp .env.example .env
   # Отредактируйте .env

   # Неправильно (не передавайте пароль в аргументах командной строки)
   python mikrotik_audit.py --ssh-pass mypassword
   ```

2. **Не коммитьте .env**
   ```bash
   # Проверьте перед коммитом
   git status
   git diff --cached
   ```

3. **Используйте SSH ключи** вместо паролей при возможности

4. **Ограничьте доступ** к файлам с отчётами

5. **Регулярно обновляйте** зависимости
   ```bash
   pip install --upgrade -r requirements.txt
   ```
